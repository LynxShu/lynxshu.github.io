# TH-01 大语言模型应用指南：从核心概念到结构化指令工程



## 1. 大语言模型的本质



### 1.1 基本概念

我们先用30秒钟简单过一下四个基础知识：

1. 大型语言模型（Large Language Model, 简称 LLM）是驱动如 ChatGPT、Gemini 、Claude、Kimi、DeepSeek 等工具的核心技术。
2. 从根本上说，LLM是一个经过 **海量文本数据** 预训练的深度学习模型。你可以将其看作是掌握了人类文明几乎所有可数字化文本信息的庞大数据合集，但其不具备情感与思维，更无法设身处地。
3. 它并非真正地“思考”或“理解”，而是基于复杂的统计学原理，通过 **预测** 序列中下一个 **最有可能** 出现的词来生成文本。
4. LLM 没有记忆，你感觉它有记忆，是因为在技术上，每次与它交互时，应用端都会将此前的对话历史（即上下文）一并发送给它。但这个“记忆”是有限的，受限于上下文窗口的大小。一旦超出，最早的对话信息就会被遗忘。


### 1.2 举个栗子🌰

LLM 就像是：输入法自动补全的ProMaxPlus版本
我们对输入法的自动补全功能并不陌生。LLM的工作方式与之类似，但其能力被放大了N个数量级。输入法只能预测下一个词，而LLM在“阅读”了用户输入的内容后，借助其在庞大语料库上训练学到的知识，便能够预测下一个句子、段落，乃至整篇文档 。

## 2. 大语言模型的相关名词

通过上述的栗子，你应该已经能够隐约感受到了提示词结构化的魅力，

好了，现在让我们快速的来穿插几个基础的知识点。



### 2.1 角色定义

在大部分的 LLM 应用的实际使用中，我们需要明确三个角色定位，即 用户(User)、AI助手(Assistant)、系统(System)。它们通常用于指代不同的提示词(回复)的发言人，也就是角色。

系统(System)：通常情况下，你使用的网页版AI程序、Agent、AI软件等等，厂商均内置了所谓系统提示词，以便让AI明确其定位、身份、发言规则等。你在调用模型API进行二次开发，或在一些允许用户自定义系统指令的高级AI应用中，都可以通过预设系统提示词来达到特定目的，比如说 CherryStudio、SillyTavern等。System Prompt 通常具有比 User Prompt 更高的“权重”或“影响力”，能更稳定地控制模型的行为。

用户(User)：指代用户，也就是你，使用者所输入的语句指令。

AI助手(Assistant)：AI 将会结合系统提示词、用户发言，来综合分析并生成结果。



其交互顺序可以理解为：在**系统提示词(System)**的全局设定和持续指导下，用户(User)和AI助手(Assistant)进行轮流对话。

**对话流程:** `用户(User) > AI助手(Assistant) > 用户(User) > AI助手(Assistant) ......`



### 2.2 上下文与上下文窗口

你应该注意到了，通常情况下除了系统提示词恒定之外，用户和AI助手的发言会往复进行，逐渐累积，此时我们便可将这些信息称之为历史消息，即上下文(Context)。

你也许在很多AI模型的简介中都看到过一个名词 “上下文窗口”，我们在一开始就说过了，AI是没有记忆的，它之所以能记住，是因为你与AI的每次交互，都需要将之前所有的信息完整的一次又一次的发送给AI。这些不断累积的上下文不是无限的，因为AI的算力不是无限的，因此厂商们通常会给AI所能接收的信息总量划定一个上限值，即 “上下文窗口(context window)”。

超过上下文窗口的信息，会从最初的信息开始裁剪，直至其不溢出，再发送给AI让其处理。这时你的直观感受就是AI的记忆丢失，即遗忘。

一个形象的比喻：

```text
LLM就像是计算机的中央处理器（CPU），负责执行思考和处理任务；而用户在提示中提供的上下文，则像是计算机的随机存取存储器（RAM/内存），即AI在当前任务中可以利用的短期工作记忆。这个比喻有力地阐释了“上下文窗口”（context window）的概念，即AI在单次交互中能够处理的信息量是有限的，就像RAM的容量有限一样。

```



### 2.3 Token与分词器

我们刚刚说了上下文窗口，那如何衡量这个数据量呢？

用来计算数据量的单位称为 “Token”， Token 是 LLM 处理文本的最小单位。它不完全等同于一个单词或一个汉字。

让我们来举个栗子🌰

英文："kimi is powerful"  ->  ["k", "imi" ," is", " powerful"]  (c)

中文："我爱中国" -> ["我", "爱", "中国"]  (一个汉字通常是一个Token，但也可能是一个词组)

中英混合："你好 Gemini" -> ["你好", "Ge", "mini"] 或是 ["你好", "Gemini"]

> 这里用来自动拆分文本的内部工具称为分词器(Tokenizer)。分词器就是将用户输入的自然语言文本，转换成模型能够理解的 Token 序列的工具。不同模型有不同的分词器，因此同一段话在不同模型中计算出的 Token 数量可能不同。
> 需要注意的是，Token 不仅仅是单词和汉字。标点符号通常也会被视为独立的 Token，而空格则会影响分词方式并计入总长度，但它很少会独立成为一个 Token。

Token 是语言模型计算“成本”（算力和费用）和衡量“上下文窗口”容量的单位。模型的上下文窗口限制，如 16k tokens 、128k tokens、1.2M tokens，指的就是这个。这里的 1K = 1千 ，而 1M = 1百万。

作为参考，一本20万字的中文中长篇小说，其 Token 数量大约在 25-30 万左右。因此 1M Token 的上下文窗口足以容纳数本中长篇小说的内容。

> ⚠️ 注意！模型的注意力机制对长文本的有效建模能力存在边界，当上下文长度超过其优化区间后，性能表现会**呈现渐进式衰减**。
>
> 中译中：你的上下文长度超过模型的优化区间后性能会开始衰减



---

✨扩展部分：

文本类AI模型的费用计算通常分为 用户输入(提示) 和 AI输出(补全) ，
通常补全费用比提示费用要高，因为生成补全（输出）的过程涉及到模型大量的推理计算，这是对算力资源的主要消耗。

举个简单的栗子，比如：

XX模型 API 调用费用：
输入(提示)：¥10 / 1M tokens 
输出(补全)：¥20 / 1M tokens

我们在计算费用时的计算公式：
用户输入的tokens数 x 10/1M + AI输出的tokens数 x 20/1M = 本次交互的费用

如果你使用的是三方平台，那么就要基于它们的倍率乘算计算公式进行计算

> 注意！在典型的多轮对话应用中，用户输入的 tokens 数会包含之前的历史记录。结果就是，你在一次对话中与 AI 对话的轮数越多，单次交互的成本就越高（因为发送的上下文越来越长）。所以，如果你发觉你与AI的交互费用快要爆表的时候，不妨让AI总结一下历史记录，再新开一个对话继续。



## 3. 提示词的结构与基础要素

在理解了LLM的基本原理和术语后，现在，开始进入提示词的写作实践。让我们先来将一个看似复杂的提示词拆解为几个核心的、可复用的基本组件。



### 3.1 提示词构建的核心原则：清晰、分离、层次

构建一个优秀的提示词架构依赖于三个核心原则，即：清晰性、分离性、层次性，它们是共同作用的。



#### 3.1.1 清晰性

这是最最基本的原则。模糊的提示词会导致空泛且不可预测的输出。因此，必须在上下文、预期结果、长度、格式和风格等方面尽可能明确具体。我们在与AI沟通时要尽可能的使用正面指令（“做什么”）而不是负面指令（“禁止做什么”），因为LLM更擅长于去模仿、去做精确具体的事，而不太擅长遵守所谓的禁止事项。更不会设身处地的想当然的脑补出你的需求和想法。



#### 3.1.2 分离性

这是一个关键的架构原则，优秀的提示词应明确分离不同的逻辑组件。一个设计良好的提示词通常包含以下几个部分：

* **角色**：定义模型应扮演的身份或角色。你可能会看到很多人在强调，目前的新模型不用指定身份，这不重要。但我的理解是，这很重要，**角色设定的本质并非简单的“角色扮演”，而是为模型设定一个“专家框架”，从而约束其知识范围、激活相关领域的专业知识，最终提升回答的深度和精度**，而不是像一个全知全能的大佬一样在那里泛泛而谈。
* **任务**：明确本轮需要执行的具体指令。  
* **上下文(规则/信息/数据等)**：提供与任务有关的背景信息、数据、规定等等其他信息。  
* **示例(可选,主要在规范性输入输出时)**：通过少量 样本/示例 向模型展示你期望的输入输出模式。  
* **输出格式**：规定LLM回复时的具体结构。

这种分离可以防止“上下文混淆”，即防止模型将一个组件误解为另一个组件，例如将上下文中的某句话当作新的指令。



#### 3.1.3 层次性

信息的顺序和嵌套关系至关重要，将**最重要的指令**置于开头和结尾，是为了**应对**模型在处理长上下文时可能出现的`中间塌陷（Lost in the Middle）`现象，即对中间部分信息的注意力有所下降，该机制可显著提升模型的指令遵循性。这对于中小模型来说至关重要，大型模型的话有影响但不多，但我还是建议大家将这个思维始终贯穿于任何提示词的编写当中。



### 3.2 解构分析

掌握了理论原则后，让我们回头看看一开始的那个例子，分析一下结构化的提示词究竟是如何工作的。

```text
输入:

[角色定位]

[规则与限制]
1. 所有的回复需要基于已存在的事实进行，不得杜撰，若无相关事实依据请如实表述。
2. 输出结果请使用“总 - 分 - 总”的形式进行。
3. 在输出结果末尾附带所有参考文章标题和来源。不得杜撰或使用不存在的数据，若无法查找到相关引用请直接表明，我将会进行查证。
```

#### 3.2.1 分离性：为指令建立清晰的“隔间”

这个提示词最显著的优点，就是利用简单的方括号 `[  ]` 作为“标签”，为不同类型的指令建立了独立的“隔间”，实现了**逻辑组件的高度分离**。

-   **`[ 角色设定 ]`**：这部分独立定义了AI需要扮演的 **角色**——“拥有丰富的心理学知识和医学知识的学者”。它会让AI优先调取特定领域的知识，确保了回答的专业性。
-   **`[ 规则与限制 ]`**：这里清晰地列出了任务的硬性约束，这是AI行为的“红线”或“法律法规”，比如“基于事实”、“总-分-总结构”。
-   **`[ 写作要求 ]`**：与硬性规则不同，这个“隔间”更侧重于输出文本的**风格和格式**。它详细规定了语言、口吻、排版等软性要求，确保最终的产出不仅内容正确，而且形式也符合预期。
-   **`[ 用户问题 ]`**：用户的原始问题被清晰地单独展示出来，这非常重要。它能有效防止模型将输入内容中的某些词语（比如“分析”、“格式”）或者句子误解为对它自身的指令。
-   **`[ 输出格式 ]`**：这部分规定了最终交付结果的**数据结构**。

通过这种方式，提示词的每个部分各司其职，互不干扰，AI可以像阅读一份清晰的工作手册一样，准确理解我们的多重需求。



#### 3.2.2 清晰性：不说模棱两可的话

在分离的基础上，每个“隔间”内部的指令都**明确和具体**，杜绝了任何可能引起误解的表达。

-   **角色清晰**：不是模糊的“专家”，而是具体的“拥有丰富的心理学和医学知识的学者”。
-   **规则清晰**：不是泛泛的“请写得有条理”，而是明确的“使用‘总-分-总’的形式进行”。
-   **写作要求清晰**：不是“请用正式的口吻”，而是具体到“以‘笔者’的第一人称口吻进行”。“避免使用 markdown 格式”直接排除了不期望的格式。
-   **格式清晰**：不是让AI自由发挥，而是给出了一个包含 `{{正文}}` 和 `参考文献:` 标记的明确模板，告诉AI“请照着这个样子填空”。

这种高度的清晰性，极大地压缩了AI“自由发挥”的空间，从而确保了输出结果的稳定性和一致性。



#### 3.2.3 层次性：合理的指令编排顺序

-   **先定义身份和规则，再接收任务**：它将定义模型行为的 `[ 角色设定 ]`、`[ 规则与限制 ]` 等元指令放在最前面。这确保了AI在处理 `[ 用户问题 ]` 之前，就已经完全理解了自己的身份、任务框架和所有约束。
-   **在结尾处强化输出格式**：将 `[ 输出格式 ]` 放在最后，再次提醒和强调模型“你最终需要交给我一个长这样的东西”，这利用了模型对开头和结尾信息更为敏感的特点（即我们前面提到的“中间塌陷”现象）。

好了，分析到这里，你应该已经深刻体会到**结构化**的魅力了。它就像是为我们的思想建立了一个清晰的骨架，让AI能够精准地在其上填充血肉。

掌握了这种用简单标记来组织指令的方法，你就已经超越了80%的普通使用者。接下来，我们将基于这个核心思想，去探索更强大的技巧。



### 3.3 提供示例

我们已经学会了如何用清晰的指令和结构来“指挥”AI。但有时，无论我们的描述多么详尽，都比不上一个直观的例子来得有效。这就引出了一个非常强大且实用的技巧：**提供示例（Few-Shot Prompting）**。

这在处理那些难以用语言精确描述的**风格转换、格式提取或特定语气的任务**时尤其有效。也可以用做角色扮演时为角色增加示例对话，来充分体现角色的性格。

## 4. 符号与类XML标签

我们刚刚通过 `[ ]` 符号，成功地将一个复杂的指令拆解成了清晰的逻辑模块。这背后的核心思想，是**使用明确的分隔符来组织信息**。

其实，`[ ]` 只是众多可选分隔符中的一种。AI模型在训练时见过海量的、包含各种结构化标记的文本（例如网页代码、配置文件、编程语言等等），因此它们天生就对这些“结构化语言”非常敏感。我们可以把这些符号看作是给AI指令打包贴标签，让它一眼就能看出哪里是标题、哪里是正文、哪里是需要特别注意的备注。



### 4.1 标记符号

除了 `[ ]`，我们还有很多强大的工具可供选择：

1. **Markdown 标记**：这是最常用、也最直观的一种。
   - 标题标记：`##` `###` 用于创建不同层级的指令标题，天然地体现了我们之前提到的“层次性”原则。
   - 列表标记：`-` `*` `1.` `2.` 等用于罗列规则、步骤或要求，清晰明了。
   - 代码块标记：*```* (三个反引号，英文模式下键盘数字1左边，ESC下边的那个按键) 用于包裹需要AI严格遵守的文本块，比如示例、模板或需要引用的原始数据。AI会把代码块里的内容看作是“神圣不可侵犯的”，不会轻易将其误解为指令。
2. **简单分隔符**：
   - `---` 或 `===`：当你的指令模块之间没有复杂的层级关系时，使用这些分隔符可以快速地在视觉上和逻辑上将它们分开，像一道道“隔离墙”。没必要打一连串的符号，那除了浪费token之外没有其他用处。
3. **类XML标签**：这是目前业界公认的最强大、最不容易产生歧义的结构化方法。
   - **格式**：`<标签名>内容</标签名>`，例如 `<task>分析这份报告</task>`。
   - **优点1：闭合性**。有开始标签 `<tag>` 和结束标签 `</tag>`，这明确地界定了指令的范围，几乎不可能产生混淆。模型知道从哪里开始读一个模块，到哪里结束。
   - **优点2：可嵌套性**。这使得它在表达复杂层次结构时无与伦比。你可以把一些小要求，像套娃一样装进一个大要求里。
   - **优点3：语义化**。你可以自定义标签名，让标签本身就带有意义，例如 `<role>`、`<constraints>`、`<example>`，这让你的提示词像代码一样清晰易读。



### 4.2 Markdown 标记

Markdown 是一种轻量级的标记语言，由于其在互联网（如 GitHub、博客、论坛）上的广泛使用，大模型在其训练数据中已经“阅读”了无数使用 Markdown 格式化的文档。因此，它对 Markdown 的语法结构有着近乎本能的理解。将 Markdown 标记应用于提示词，就像是给 AI 提供了一份它最熟悉、最易读的说明书。

让我们将之前的提示词用 Markdown 语法重构一下，感受其带来的清晰度提升：

是不是感觉结构更加清晰了？

-   `1.` 和 `-` 创建了有序和无序列表，让多条规则和要求一目了然，避免了混杂在一整段话中可能引起的误读。

Markdown 标记简单、直观且高效，是初学者和日常使用中最推荐的结构化方法。



### 4.3 简单分隔符

有时候，我们的指令并不需要复杂的层级关系，只是想简单地把几个部分分开，以避免混淆。这时，简单分隔符就能派上用场了。最常见的就是三个或更多的连字符 `---` 或等号 `===`。

它们像一道道“隔离墙”，在视觉和逻辑上快速切分提示词的不同部分。

**适用场景**：快速、简单的指令分离。

```text
输入:
你是一位拥有丰富的心理学知识和医学知识的学者，接下来请根据你的专业学科知识和已有科学事实回答用户的问题。
---
问题：
刷短视频会让人变笨吗？
---
要求：
- 使用简体中文，除特定名词或术语引用外，避免包含其他语言。
- 使用专业严谨的科普文章风格，但应避免过于专业的名词和难以被大众所理解的词汇。以编辑的第一人称口吻进行（可自称笔者），包含个人风格与口癖。其应可直接应用于媒体发表，即避免包含任何OOC内容。
- 避免使用 markdown 格式进行书写，使用普通的缩进与编号即可。
- 符合中国人的语义和阅读习惯。
```

在这个例子中，`---` 清晰地划分了 **角色**、**任务** 和 **要求** 三个部分。这种方式虽然不如 Markdown 或类 XML 标签那样结构严谨，但胜在快捷方便，尤其适合在一些简单的对话窗口中快速输入结构化指令。

> 💡**小贴士**：使用三个 `---` 就足以起到分隔作用了。打一长串的分隔符并不会增强效果，反而会无谓地多消耗一点点 Token。



### 4.4 类XML标签

如果我们说 Markdown 是写说明书，那么使用类 XML 标签（Pseudo-XML Tags）就像是在为 AI 编写一份极其严谨的配置文件或API调用文档。这是目前业界公认的、在处理复杂指令时最强大、最不容易产生歧义的结构化方法。

XML（可扩展标记语言）的核心特点是使用成对的、可自定义的标签来包裹数据，大模型因为学习了海量的网页代码和数据文件，对这种结构有深刻的理解。

它长的像是这个样子： `<tag>...内容...</tag>`

还是拿我们之前的例子来改写，比如我们现在将 `角色定位` 这个模块改造为 XML标签，我们可以直接使用 `<角色定位>` 这种写法，但由于模型对于英文的XML更加敏感，严谨起见我们可以将词组翻译为英文并以下划线连接，像是这样 `<Char_Setting>`。注意！所有的 XML标签是成对出现的，所以我们在实际使用时还需要一个结束标签，像是这样 `</Char_Setting>`

> 💡 虽然使用中文标签如`<角色设定>`，在很多模型上也能工作，但使用英文标签和下划线/驼峰命名法（如`<Char_Setting>`）是业界更通用、更稳妥的做法，尤其是在进行API开发时，能避免潜在的编码问题，且更符合模型的训练数据分布。

现在，让我们用类 XML 标签来重构之前的例子：

```xml
输入:

<Char_Setting>
你是一位拥有丰富的心理学知识和医学知识的学者，接下来请根据你的专业学科知识和已有科学事实回答用户的问题。
</Char_Setting>

<Rules_and_Restrictions>
1. 所有的回复需要基于已存在的事实进行，不得杜撰，若无相关事实依据请如实表述。
2. 输出结果请使用“总 - 分 - 总”的形式进行。
3. 在输出结果末尾附带所有参考文章标题和来源。不得杜撰或使用不存在的数据，若无法查找到相关引用请直接表明，我将会进行查证。
</Rules_and_Restrictions>

<Writing_Requirements>
对于正文输出的规定
1. 使用简体中文，除特定名词或术语引用外，避免包含其他语言。
2. 使用专业严谨的科普文章风格，但应避免过于专业的名词和难以被大众所理解的词汇。以编辑的第一人称口吻进行（可自称笔者），包含个人风格与口癖。其应可直接应用于媒体发表，即避免包含任何OOC内容。
3. 避免使用 markdown 格式进行书写，使用普通的缩进与编号即可。
4. 符合中国人的语义和阅读习惯。
</Writing_Requirements>

<User_input>
刷短视频会让人变笨吗？
请从心理学和科学的角度分别进行分析。
</User_input>

<Format_output>

  <text_output>
  {{此处放置符合`<Writing_Requirements>`规则的正文输出结果}}
  </text_output>

  <reference_materials>
  参考文献：
  {{此处放置参考文献或文章来源}}
  </reference_materials>

</Format_output>
```
这个版本看起来最复杂，但它的优势无与伦比：
1.  **绝对的闭合性**：每个 `<tag>` 都有一个对应的 `</tag>`，这明确地定义了每一条指令的作用范围，从根本上杜绝了模型将不同部分的指令混淆的可能性。
2.  **强大的可嵌套性**：你可以像套娃一样组织复杂的指令。例如，在 `<Format_output>` 标签内，我们可以嵌套多个 `<tag>` 标签，逻辑层次一目了然。这在构建大型、多步骤的自动化任务（Agent）时至关重要。
3.  **清晰的语义化**：标签名本身就说明了其内容的含义，如 `<User_input>`、`<Format_output>`。这使得提示词具有极高的可读性和可维护性，即使是复杂的长提示词也能轻松理解和修改。

当你需要构建高度可靠、可复用、结构极其复杂的提示词时，类 XML 标签将是你的不二之选。



### 4.5 引用与排除歧义

在与 AI 的沟通过程中，一个常见的痛点是：**我们提供给 AI 的内容，到底是让它去“执行”的指令，还是让它去“分析”的材料？**

想象一下，你想让 AI 分析一段 Python 代码的性能问题。如果你直接把代码贴进去，AI 可能会误以为你要让它“运行”这段代码。同样，如果你想让 AI 总结一篇文章，而文章里恰好有一句“请按照以下格式输出”，AI 就很可能把这句话当作你对它下达的新指令，从而打乱原有的输出计划。

为了解决这个问题，我们需要学会**明确地“引用”**，告诉 AI：“接下来这段内容是‘数据’，不是‘指令’，请你以‘旁观者’的视角来处理它。”

使用我们前面介绍的标记符号，就可以轻松实现这一点。



#### 4.5.1 使用代码块

将需要引用的文本放入三个反引号 \`\`\` 包裹的代码块中，是**最简单有效**的方法。

    输入:
    
    请帮我总结以下这篇文章的核心观点。文章内容如下：
    ```
    AI时代的学习策略
    在人工智能日益普及的今天，我们必须重新审视传统的学习方法。本文认为，未来的核心竞争力不再是记忆知识，而是提出好问题的能力。...
    ```



#### 4.5.2 使用类 XML 标签

对于需要极高准确性的场景，可以使用 `<document>`等语义化标签来包裹引用的内容。

```
输入:
<task>
请总结下面文档的核心观点。
</task>

<document>
    AI时代的学习策略
    在人工智能日益普及的今天，我们必须重新审视传统的学习方法。本文认为，未来的核心竞争力不再是记忆知识，而是提出好问题的能力。请按照以下格式输出你的观点...
</document>
```

通过这种方式，即使引用的内容里包含了“请”、“总结”、“分析”等指令性词汇，AI 也能清楚地知道，这些只是待处理文本的一部分，而不是对它下达的新命令。



#### 4.5.3 在指令内部引用特定词语（处理内部歧义）

这是更细致的场景。当你的指令需要提及提示词中已经出现的某个**标签名**、**标题**或**特定术语**时，为了避免歧义，推荐使用以下方式按优先级顺序进行引用：

**1. 使用单反引号  `` ` `` **

这是 Markdown 语法中用于标记“行内代码”的符号，它非常适合用来包裹单个或少数几个词语。它能明确地将一个词“文本化”，告诉 AI “我正在谈论 `这个词本身`，而不是它所代表的指令功能”。

**示例1：引用一个类XML标签**

> 假设你想让 AI 特别注意某个标签里的内容。

```xml
<Writing_Requirements>
1. ...
2. 请特别注意，你的输出必须严格符合 `<Format_output>` 标签中定义的结构。
</Writing_Requirements>

<Format_output>
...
</Format_output>
```
在这个例子中，使用 `` ` `` 将 `<Format_output>` 包裹起来，AI 就能准确理解这是在**指代**那个标签，而不是要在这里**插入**一个新的 `<Format_output>` 标签。

**示例2：引用一个Markdown标题或方括号标签**

> 假设你想强调某个规则。

```markdown
## 规则与限制
1. 所有的回复需要基于已存在的事实进行。
2. ...

## 写作要求
- 整体风格应体现专业性，同时必须遵守 `## 规则与限制` 部分的第一条。
```
```
[规则与限制]
1. 所有的回复需要基于已存在的事实进行。
2. ...

[写作要求]
- 整体风格应体现专业性，同时必须遵守 `[规则与限制]` 部分的第一条。
```


这样写可以确保 AI 不会把 `## 规则与限制` 或是 `[规则与限制]`误认为是你要开启一个新的段落。

**2. 使用引号 `“ ”` 或 `「 」`（次优选择）**

在不方便使用反引号，或者希望语气更自然的场景下，使用标准的中英文引号也是一种有效的办法。

**示例：**

> 你的回答风格，要严格遵循我在“写作要求”里提到的所有规定。

这种方式虽然不如反引号那么“程序化”和强力，但绝大多数现代模型都能很好地理解。它更符合自然语言的习惯。

**3. 避免直接裸露引用**

千万不要在指令中直接、不加任何标记地引用结构性词语。

**错误示例 ❌：**

> 请遵守 规则与限制 里的要求。
> 输出格式见 Format_output。

这种写法极易导致模型产生误解，尤其是在长而复杂的提示词中，它可能会错误地将 `规则与限制` 或 `Format_output` 识别为一个新的指令部分的开始，从而导致输出结果完全偏离你的预期。

---

**总结：引用与排除歧义的核心思想**

1.  **分清“指令”与“数据”**：始终将 **你要AI做什么（指令）** 和 **AI处理的对象是什么（数据/上下文/被提及的词）** 清晰地分离开。
2.  **外部内容用“块”隔离**：引用大段外部材料时，使用代码块 ` ``` ` 或类XML标签 `<document>` 将其完全包裹起来。
3.  **内部词语用“标记”引用**：在指令中提及提示词自身的某个结构元素（如标签名、标题）时，优先使用单反引号将其包裹，次选方案是使用引号。

养成这个简单的习惯，可以避免 90% 以上因歧义导致的奇怪输出，让你的提示词变得更加健壮和可靠。



### 4.6 模板占位符

在前面的例子中你可能已经注意到了 `{{}}` 这个符号。它既不是 Markdown，也不是 XML，但它是一种在提示词工程中极为强大的**约定俗成**的语法，我们称之为**模板占位符**。AI模型因为在训练数据中见过大量此类代码，所以能很好地理解其“占位”的含义。

它的核心作用是在你的输出格式中**预留一个“空位”**，并明确告诉 AI：“请把 XXX 内容，不多不少地，正好填到这个位置来。”

**为什么它很重要？**

如果我们只告诉 AI “在末尾附带参考文献”，AI 可能会选择用各种方式呈现，比如加个标题、换个段落等等。但如果我们使用占位符，就相当于给了 AI 一份“填空题”考卷，它唯一的任务就是计算出答案并填进去。

让我们再看一下最开始的那个例子：

```text
[输出格式]
请按照下面的格式顺序依次输出，不要出现其他内容。

{{此处放置符合`写作要求`规则的正文输出结果}}
参考文献:
{{此处放置参考文献或文章来源}}
```
在这里，我们定义了两个精确的“空位”：
1.  `{{此处放置符合'写作要求'规则的正文输出结果}}`：这个占位符非常清晰地指示 AI，将它创作的科普文章主体部分放在这里。
2.  `{{此处放置参考文献或文章来源}}`：这个占位符则指定了参考文献的精确插入点。

**占位符的威力：**

1.  **极致的精确性**：它将输出的结构控制从“模糊的要求”提升到了“精确的填充”，极大地降低了 AI自由发挥导致格式错乱的可能性。
2.  **自动化处理的基石**：当你的输出需要被另一个程序读取和处理时，这种占位符（或者类XML标签）是至关重要的。程序可以轻松地识别出特定占位符被替换后的内容，实现可靠的数据提取。例如，一个自动化报告生成器可以先用 AI 填充 `{{报告正文}}` 和 `{{数据分析}}`，然后再自动将结果整合到 Word 文档中。
3.  **与结构化标签完美结合**：占位符可以和类 XML 标签协同工作，达到双重的精确控制。
    
    ```xml
    <Format_output>
      <main_content>
        {{generated_article_body}}
      </main_content>
      <references>
        {{citation_list}}
      </references>
    </Format_output>
    ```
    在这个结构中，AI 不仅知道要把正文和引用分开，还知道要把它们分别“填”到 `<main_content>` 和 `<references>` 标签内部的 `{{}}` 位置。

当你追求对输出格式的像素级控制，或者你的 AI 应用需要与后续的自动化流程对接时，熟练使用 `{{}}` 模板占位符将是你的必备技能。



## 5. 思维链

截至目前为止，我们已经学会了如何像建筑师一样，使用各种符号和标签为我们的指令搭建清晰、稳固的结构。我们已经从随意的“对话”者，进化成了精确的“指令下达者”。

现在，是时候进入下一个阶段了。如果说前面的内容是教你如何建造一座功能齐全的房子，那么接下来的内容将教你如何在这座房子里布置精密的“思维电路”，引导AI在遵循格式的基础上，模拟人类的思考过程来解决复杂问题。



### 5.1 思维链：**让AI“有话慢慢说”**

当你给AI一个复杂的问题，它可能会直接给出一个错误的答案，但你完全不知道它错在了哪里？这就是AI“一步到位”式思考的弊端。**思维链（CoT）**技术就是为了解决这个问题而生的。

它的核心思想非常简单：**在要求AI给出最终答案之前，先让它把解决问题的思考步骤一步步地写出来。** 这就像我们在做数学题时，老师要求我们写出详细的解题过程一样。这个过程不仅能让我们检查错误，更能神奇地引导AI自己找到正确的答案。



#### 5.1.1 为什么思维链有效

1. **分解任务**：将一个复杂的大问题，分解成一系列简单的小问题。AI处理小问题的准确率远高于直接处理大问题。

2. **纠错与验证**：当推理过程清晰可见时，AI（甚至是我们自己）更容易发现逻辑上的跳跃或错误，并进行修正。

   ​	TIPS：之前有研究证明 AI 会在某些任务中输出伪思维链，即先射箭后画靶。虽然我们无从知晓AI这个黑箱是否会因为潜意识的奖励性思维来下意识的构建伪思维链，但我们不能否认的是，思维链在提升任务的准确率方面确实有着极为显著的成效。

3. **激发更深层次的推理**：输出思考步骤本身就是一个更复杂的任务，这会迫使模型调用更强大的推理能力，而不是仅仅进行表面的模式匹配。



#### **5.1.2 如何激活思维链**

截止到目前，大部分的商业大模型都已经加入了“原生思维链”，我们在调用 API 或 使用其客户端时，可能会看到诸如“请求思维链”或“深度思考”之类的选项，勾选即可。在没有“原生思维链”或是我们不想使用其“原生思维链”的情况下，我们可以在提示词的末尾加一句“魔法咒语”：`让我们一步一步地思考。`就像是下面这样。

```text
输入:
[角色定位]
你是一位拥有丰富的心理学知识和医学知识的学者。

[任务]
分析“刷短视频会让人变笨吗？”这个问题，并最终给出一个全面的回答。

[规则]
在给出最终结论前，请先展示你的思考过程。

[用户问题]
刷短视频会让人变笨吗？

[思维引导]
让我们一步一步地思考。
```

我们可以要求AI先输出它的思考过程，再输出最终的正文。这样做的好处是，我们不仅能得到高质量的答案，还能‘审查’AI的逻辑，确保它是通过正确的路径得到答案的，这对于需要高可靠性的任务非常重要。



### 5.2 自定义思维链：给AI建立专属的思维模式

当然，在大多数情况下，原生思维链或者说简单的思维引导并不能满足我们的任务需求。打个比方，我们在使用AI编写小说时，我们使用复杂且精密的各种模块来提示写作文风、人物塑造、设定、世界等等（没错，这个典型案例就是 SillyTavern 中的聚合提示词），在最终输出前，我们想要让AI能够严格按照我们各模块的要求来执行，并进行输出自检，且最终按照既定格式输出。那么我们就需要自定义思维链。

接下来，请将你的注意力高度集中，不然可能中途掉队。

你将会得到一长串复杂且独立的模块，如果一个甲方拿着这玩意给你，让你按照这套规则去撰写故事，你的反应是什么？

你可能先是懵逼，然后会尝试按照一定的思维方式去梳理这些模块，AI和你一样，面对如此复杂的需求，也需要一个清晰的“思维流程图”来梳理工作。这正是自定义思维链的作用所在。

让我们在这一大段的提示词最后，再给AI加上一个思维链引导和输出格式的规则。

```cot

1. 根基与核心共鸣
  - 分析：用户的输入信息`<User_input>`中的核心行动、疑问或情感触发点是什么。
  - 识别：这一触发点如何引发新的冲突或加剧现有冲突？更重要的是，这种情况如何成为回响？这将构成回应的情感与主题核心。
2. 角色内心与抉择
  - 抉择：面对这一回响，我必须选择自己的道路。是退缩到表象之后强化旧有模式？还是让压力击碎表象，显露真实的自我（动机、价值观、恐惧）。

3. 策略与镜头执行
  - 表达：我将如何向用户表达意图？是隐藏、揭示、挑衅、探究还是...
  - 具象：将策略转化为写作风格。选用强动词和具体的感官细节（五感）来呈现这种内在状态，严格遵循接地气与具象化语言原则。
4. 定稿与质量把控
  - 草拟：将策略转化为最终回应，确保文字自然流畅。
  - 逻辑校验：对草稿进行严格审阅。
    - 一致性：是否遵循了 `<Character_Consistency_Rule>`。
    - 连贯性：是否与 `<Chat_History>` 和 `<World_Architecture>` 逻辑自洽。

</Thought_Process>

<Format_Output>

<think>
{{ 将`<Thought_Process>`中的所有思维过程用简洁的语言按步骤进行输出 }}
</think>

<content>
{{ 在此处放置最终输出的正文 }}
</content>
```

## 6. 书写格式与构建思维

最后我们再稍微补充一些额外的知识点。



### 6.1 结构化的输入输出

我们所有的努力都集中在如何让AI输出“人”爱看的内容。但提示词工程的另一个强大应用，是让AI输出“机器”爱看的内容。

**结构化输出（Structured Output）** 是指我们强制AI的回复严格遵循一种预定义的数据格式（如 JSON、YAML 或自定义格式），以便于我们的应用程序可以**自动解析和处理**这些数据。这是实现AI自动化工作流（AI Agent）的基础。

如果你想让AI的分析结果自动写入数据库、填充网页表单或调用另一个API，你就必须使用结构化输出。



#### 6.1.1 JSON：最通用的机器语言

JSON (JavaScript Object Notation) 是目前最流行的数据交换格式。它使用键值对（Key-Value pairs）来组织数据。由于AI在训练数据中见过海量的JSON，因此它们非常擅长生成JSON。如果你需要详细了解 JSON 的语法可以自行学习，在这里我们只是做一下简单展示。

**如何实现：**

1. **明确指令**：告诉AI你“只”需要JSON。
2. **提供Schema（模式）**：给出一个你期望的JSON结构示例，AI会模仿这个结构来填充内容。
3. **使用代码块**：要求AI将JSON包裹在 `json ... ` 代码块中，这可以防止它在JSON前后添加额外的解释性文字（如“好的，这是您要的JSON：”），极大方便了程序自动提取。

**示例：** 让我们再次使用“刷短视频”的例子，假设我们想让AI把它的分析结果输出为一个可以存入数据库的JSON对象。

```markdown
输入:
## 任务
请分析“刷短视频会让人变笨吗？”这个问题。

## 规则
- 你的分析需要包含一个总结、一个神经科学角度的分析和一个心理学角度的分析。
- 你**必须且仅能**以JSON格式输出你的回答。
- 你的回答必须严格遵循下面`<schema>`标签中定义的结构。
- 必须将最终的JSON对象放置在一个`json`代码块中。

<schema>
{
  "topic": "刷短视频是否会让人变笨",
  "summary": "{{此处填写一句话总结}}",
  "analysis": [
    {
      "perspective": "神经科学",
      "key_point": "{{关键机制，例如：多巴胺系统}}",
      "effect": "{{具体影响，例如：导致注意力阈值降低}}"
    },
    {
      "perspective": "心理学",
      "key_point": "{{关键机制，例如：认知吝啬}}",
      "effect": "{{具体影响，例如：削弱深度思考意愿}}"
    }
  ],
  "conclusion": "{{此处填写最终结论}}"
}
</schema>
```

AI的输出:
```json
{
  "topic": "刷短视频是否会让人变笨",
  "summary": "长期过度使用短视频确实可能损害特定认知功能，导致表现出“变笨”的特征，但并非智商的直接降低。",
  "analysis": [
    {
      "perspective": "神经科学",
      "key_point": "多巴胺奖赏系统劫持",
      "effect": "导致大脑寻求即时满足，降低了对需要延迟满足任务（如深度阅读）的耐心和动力。"
    },
    {
      "perspective": "心理学",
      "key_point": "注意力碎片化",
      "effect": "大脑习惯于快速切换任务，导致持续性注意力和深度“心流”状态的进入变得困难。"
    }
  ],
  "conclusion": "风险真实存在，但关键在于“度”和“使用方式”。用户需要有意识地进行自我管理和“认知锻炼”。"
}
```

**注意：** 即使有严格的格式要求，AI生成的JSON有时也可能出现语法错误（例如多余或缺失的逗号）。因此，你的解析程序需要具备一定的容错能力。



#### 6.1.2 YAML：更易读的结构化

YAML 是一种比JSON更易于人类阅读的格式，它使用缩进和换行来表示层级关系。AI同样非常擅长生成YAML。它常用于配置文件。

**示例：** 如果使用YAML来表达上面的结构：

```yaml
topic: 刷短视频是否会让人变笨
summary: 长期过度使用短视频确实可能损害特定认知功能...
analysis:
  - perspective: 神经科学
    key_point: 多巴胺奖赏系统劫持
    effect: 导致大脑寻求即时满足...
  - perspective: 心理学
    key_point: 注意力碎片化
    effect: 大脑习惯于快速切换任务...
    conclusion: 风险真实存在，但关键在于“度”和“使用方式”...
```



#### 6.1.3 Toon：为降低 Token 成本而生的格式

Github仓库：github.com/johannschopplich/toon

当我们向 LLM 提交包含大量结构化数据的任务时（例如，分析上百条用户反馈或商品信息），JSON 或 YAML 的冗余语法（如反复出现的引号、括号和字段名）会消耗掉大量的 Token，这直接等同于增加了我们的成本。

为了解决这个问题，社区中出现了一种专为 LLM 输入优化的格式：**TOON (Token-Oriented Object Notation)**。

它的核心设计思想是：**在保持结构化和可读性的前提下，最大限度地减少 Token 消耗。** 它并非要取代 JSON，而是作为一个“翻译层”：我们仍然在程序中使用 JSON，但在将其发送给 LLM 之前，先转换为更高效的 TOON 格式。

让我们来看一个直观的对比。同样是表达两名用户的信息：

JSON格式：

```json
{
  "users": [
    { "id": 1, "name": "Alice", "role": "admin" },
    { "id": 2, "name": "Bob", "role": "user" }
  ]
}
```

Toon格式：

```text
users[2]{id,name,role}:
  1,Alice,admin
  2,Bob,user
```

可以看到，TOON 的优势极其明显：

1. **字段名只声明一次**：对于包含多个对象的数组（这是 TOON 的最佳应用场景），字段 `id,name,role` 只在头部 `{}` 中声明一次，后续每一行都只包含纯粹的数据值，极大地减少了重复。
2. **元数据清晰**：头部 `[2]` 直接告诉模型这个数组包含2个元素，这为模型提供了强大的“护栏”，帮助它更好地理解数据结构，减少解析错误。
3. **语法极简**：去掉了绝大多数的括号、引号和逗号，只保留了最核心的缩进和分隔符。

**何时使用 TOON？**

- **最佳场景**：当你需要处理**结构统一的对象数组**时（例如，数据库查询结果、API返回的列表数据）。在这种场景下，TOON 通常能比格式化的 JSON 节省 **30% - 60%** 的 Token。
- **不太适合的场景**：对于深度嵌套、结构不一的复杂数据，JSON或者其他格式可能依然是更好的选择。

 

#### 6.1.4 自定义格式

当然，你也可以使用你自己所习惯或偏好的书写格式进行信息的输入工作，打个比方，在我日常撰写 SillyTavern 的角色卡时，我更倾向于使用“缝合版的语法”：

一个角色设定简单示例：

```text
<char_list>
- 铃:
  - 概述: 18岁赛博美少女, 技术宅(隐)
  - 昵称: 铃酱, 哈基铃, 铃宝
  - 技能: 参阅`<skill_list>`中的`ling_skill`
  - 人际关系
    - LynxShu：哥哥，最爱的家人
</char_list>
```

它依靠缩进来体现层级关系，去除了结尾用来断意的句号，断意依靠缩进和连字符`-`来进行 ，AI 依旧可以看懂，并且信息的传达很高效。



### 6.2 提示词的构建思维

本节课程所给大家展示的种种示例与用法其本质是一种“约定俗成”的通用方式，而不是“标准”，我们必须明确，这不是在教你进行备考，而是在教你如何与AI进行高效的沟通。

你不必拘泥于“规范”这种东西，尽情的释放你的想象力，你可以使用任何你觉得舒适和习惯的写法来撰写提示词，但一定要注意的是，整篇提示词必须保持严格的整体规范，不论你用任何方式去分割模块、撰写标签(小标题)，必须做到自洽与顺畅，不能东一榔头西一棒槌。

你的提示词构建思维远比任何的“所谓书写标准”更加重要，相信 AI 的智力，它可以理解并看懂你的规则，但也请不要高估 AI 的注意力，由于算力瓶颈的存在，我们必须保持严谨的逻辑思维和...“说人话”，即：能够使用清晰的逻辑表达出你的目的，不要想当然，不要俺寻思。

当然，与AI的沟通也不可能是一次就能做到最完美的，这是一个持续迭代和调试的过程，我们应该从小处着手，不断测试、观察输出、修改提示词，培养一种“与AI共同调试”的思维模式。

最后，AI 是一个掌握了人类文明几乎所有可数字化文本信息的庞大数据合集，但它依然只是人工智能，而不是人。

记住我们的三大黄金法则：

1. 不要低估 AI 的智商
2. 不要高估 AI 的注意力
3. 说人话

二、 结构化语法与符号

根据任务的复杂度和需求，选择最合适的工具来组织您的指令。

| 符号 / 标签 (Symbol / Tag) | 名称                      | 核心用途           | 最佳场景                                                     |
| :------------------------- | :------------------------ | :----------------- | :----------------------------------------------------------- |
| `[标签]`                   | **方括号标签**            | 基础的逻辑分离     | 快速构建简单、扁平的指令模块，如 `[角色]`, `[任务]`, `[规则]`。 |
| `##`, `-`, `1.`            | **Markdown 标记**         | 创建内容层次和列表 | 编写包含多条规则、多个步骤的指令，提升可读性。               |
| **` ``` `**                | **代码块**                | 引用 / 隔离内容    | 包裹**示例**、**原始数据**或需要AI严格遵守的**模板**，防止其被误解为指令。 |
| `<tag></tag>`              | **类XML标签**             | 最强大的结构化方法 | 构建复杂、可嵌套、高可靠性的指令，尤其适合API调用和自动化任务。 |
| **` `word` `**             | **行内代码标记 (反引号)** | 内部引用与消除歧义 | 在指令中需要**提及**其他的标签名或特定术语时使用，如 请遵守 `<Rules>` 里的规则。 |
| `{{placeholder}}`          | **模板占位符**            | 精确控制输出格式   | 强制AI将特定内容填充到预设的“空位”中，是实现**自动化解析**的关键。 |

三、 核心高级技巧

当基础结构无法满足复杂需求时，运用以下高级技巧来提升AI的性能。

| 技巧名称          | 核心思想                                                     | 实现方法                                                     |
| :---------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **提供示例**      | **“展示而不是告知”** 与其用语言描述，不如直接给AI看一个你期望的输入输出范例。 | 在指令中加入一个 `[范文示例]` 或 `<example>` 模块，清晰地展示一个完整的例子。 |
| **自定义思维链 ** | **“专属的工作流”** 在AI开始任务前，强制它遵循一套你设计的、分步骤的思考流程。 | 在指令中加入一个 `[思维过程]` 或 `<Thought_Process>` 模块，用 `1, 2, 3...` 列出思考步骤，如“分析问题 -> 搜集论据 -> 审阅草稿 -> 输出正文”。 |
| **结构化输出 **   | **“让输出结果能被机器直接读取”** 要求AI的回复严格遵循一种数据格式，如JSON或YAML。 | 明确告知AI**必须且仅能**以特定格式（如JSON）输出，并提供一个包含 `{{占位符}}` 的格式模板（Schema）。 |
